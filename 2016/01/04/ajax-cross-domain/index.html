AJAX之跨域

  • AJAX
  • 前端

导读：跨域的一些基本问题和用JSONP的解决办法

什么是跨域？

浏览器出于安全方面的考虑，不允许JavaScript跨域调用其他页面的对象，这就是浏览器的同源策略。
比如来说：
对于http://www.a.com/1.html 来说

  • http://www.a.com/2.html 与其是同源的；
  • http://www.a.com/comment/2.html 与其是同源的；
  • http://www.b.com/2.html 与其是不同源的。 原因：域名不同；
  • http://www.a.com/2.html 与其是不同源的。原因：同一个域名，但是二级域名不一样；
  • https://www.a.com/2.html 与其是不同源的。原因：协议不同；
  • http://www.a.com:8080/2.html 与其是不同源的。原因：端口不同；
  • http://sub.a.com/2.html 与其是不同源的。原因：主机不同。
  • http://70.32.92.74/2.html 与其是不同源的。原因：在跨域问题上，域仅仅是通过“URL的首部”来识别而不会去尝试判断相同的ip地址对应着两个域或两个域是否在同一个ip上。

所以在a.com 域名下的js无法操作b.com或其他域名下的对象，只能操作二级域名、端口、协议必须与主页面完全相同，否则就算跨域。

为什么要跨域？

<script type="text/java script"src="https://aja x.googleapis.com/aj ax/libs/jquery/1.4.2/jquery.min.js"></script>

上面代码就是跨域去加载Google的jQuery，这样做的好处就是加载时间少了，而且减少了服务器的流量。
其实跨域这东西很常见，把网站的一些脚本、图片或其他资源放到另外一个站点，这些操作都是跨域。

在浏览器中，<script>、<img>、<iframe>和<link>这几个标签（我发现凡是拥有”src”这个属性的标签都拥有跨域的能力）是可以加载跨域（非同源）的资源的，并且加载的方式其实相当于一次普通的GET请求，唯一不同的是，为了安全起见，浏览器不允许这种方式下对加载到的资源的读写操作，而只能使用标签本身应当具备的能力（比如脚本执行、样式应用等等）。默认情况下，跨域的URL是无法通过Ajax访问的。

有时候我们需要的不仅仅是一些脚本、图片这样的资源，我们也会希望从另外的站点调用一些数据，例如我希望获取一些blog的RSS来生成一些内容，再或者说我在“人人开放平台”上开发一个应用，需要调用人人的数据。
然而，很不幸的是，用ajax请求一个第三方的地址是被浏览器所禁止的，直接用XMLHttpRequest来Get或者Post行不通。
但是，天无绝人之路，有个叫JSONP的技术就是来解决这种问题的。原理是在远程服务器上设法把数据装进js格式的文件里，供客户端调用和进一步处理。下面我们就来讨论如何解决跨域问题。

怎么解决跨域问题？

解决跨域问题有很多办法，八仙过海，各显神通。接下来就将介绍一种很常见的解决跨域的办法：JSONP

什么是JSONP?

看到这个专有名词，可能会想到JSON，虽然JSON和JSONP只有一个字母的差别，但其实他们根本不是一回事儿。
JSON是一种数据交换格式，而JSONP是一种依靠开发人员的聪明才智创造出的一种非官方跨域数据交互协议。
可以这么来讲吧json 是一种数据格式，jsonp 是一种数据调用的方式。
上文讲到<script>标签没有跨域限制的，所以我们可以利用这个“漏洞”来达到第三方通讯的目的。当需要通讯时，就可以创建一个<script>元素，地址指向第三方API网址，形如：<script src="http://www.example.net/api?param1=1&param2=2"></script>并提供一个回调函数来接收数据（函数名可约定，或通过地址参数传递）。第三方产生的响应为json数据的包装。（所以称之为jsonp，即json padding）包装形如：callback({"name":"hax","gender":"Male"})这样浏览器会调用callback函数，并传递解析后json对象作为参数。可在callback函数里处理所传入的数据。

JSONP如何工作的？

接下来我将循序渐进的来解释JSONP的原理
上文说到<script>标签是可以加载跨域（非同源）的资源的
远程服务器a.com根目录下有个test.js文件代码如下：

alert('我是远程文件');

本地服务器local.com下有个localtest.html页面代码如下：

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">
<html xmlns="http://www.w3.org/1999/xhtml">
<head>
    <title></title>
    <script type="text/javascript" src="http://a.com/test.js"></script>
</head>
<body>

</body>
</html>

运行localtest.html页面后，毫无疑问会弹出“我是远程文件”的提示窗口，表示跨域调用成功。

现在我们在localtest.html页面定义一个函数，通过远程js来调用这个函数，并且传入参数。
localtest.html页面代码如下：

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">
<html xmlns="http://www.w3.org/1999/xhtml">
<head>
    <title></title>
    <script type="text/javascript">
    var localHandler = function(data){
        alert('我是本地函数，可以被跨域的test.js文件调用，远程js带来的数据是：' + data.result);
    };
    </script>
    <script type="text/javascript" src="http://remoteserver.com/remote.js"></script>
</head>
<body>
</body>
</html>

test.js文件代码如下：

localHandler({"result":"我是远程js带来的数据"});

运行localtest.html页面后查看结果，页面成功弹出提示窗口，显示本地函数被跨域的远程js调用成功，并且还接收到了远程js带来的数据。

很神奇，跨域远程获取数据的目的基本实现了，但是又一个问题出现了，我怎么让远程js知道它应该调用的本地函数叫什么名字呢？（这里是localHandler()）毕竟是jsonp的服务者都要面对很多服务对象，而这些服务对象各自的本地函数都不相同啊？
接下来我们解决这个问题：
现在只要服务端提供的js脚本是动态生成的就行了，这样调用者可以传一个参数过去告诉服务端需要的数据并告诉服务器我的回调函数叫什么名字，于是服务器就可以按照客户端的需求来生成js脚本并响应了。
localtest.html页面的代码：

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">
<html xmlns="http://www.w3.org/1999/xhtml">
<head>
    <title></title>
    <script type="text/javascript">
    // 得到航班信息查询结果后的回调函数
    var flightHandler = function(data){
        alert('你查询的航班结果是：票价 ' + data.price + ' 元，' + '余票 ' + data.tickets + ' 张。');
    };
    // 提供jsonp服务的url地址（不管是什么类型的地址，最终生成的返回值都是一段javascript代码）
    var url = "http://flightQuery.com/jsonp/flightResult.aspx?code=CA1998&callback=flightHandler";
    // 创建script标签，设置其属性
    var script = document.createElement('script');
    script.setAttribute('src', url);
    // 把script标签加入head，此时调用开始
    document.getElementsByTagName('head')[0].appendChild(script); 
    </script>
</head>
<body>
</body>
</html>

这次没有把远程js文件写死，而是编码实现动态查询，而这也正是jsonp客户端实现的核心部分，本例中的重点也就在于如何完成jsonp调用的全过程。我们看到调用的url中传递了一个code参数，告诉服务器我要查的是CA1998次航班的信息，而callback参数则告诉服务器，我的本地回调函数叫做flightHandler，所以请把查询结果传入这个函数中进行调用。
OK，服务器很聪明，这个叫做flightResult.aspx的页面生成了一段这样的代码提供给localtest.html
（服务端的实现这里就不演示了，与你选用的语言无关，说到底就是拼接字符串）：

flightHandler({
    "code": "CA1998",
    "price": 1780,
    "tickets": 5
});

我们看到，传递给flightHandler函数的是一个json，它描述了航班的基本信息。运行一下页面，成功弹出提示窗口
到这里为止的话，相信你已经能够理解jsonp的客户端实现原理了吧？剩下的就是如何把代码封装一下，以便于与用户界面交互，从而实现多次和重复调用。

jQuery如何实现jsonp调用？

这里依然沿用上面那个航班信息查询的例子，假定返回jsonp结果不变：

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">
 <html xmlns="http://www.w3.org/1999/xhtml" >
 <head>
     <title>Untitled Page</title>
      <script type="text/javascript" src=jquery.min.js"></script>
      <script type="text/javascript">
     jQuery(document).ready(function(){ 
        $.ajax({
             type: "get",
             async: false,
             url: "http://flightQuery.com/jsonp/flightResult.aspx?code=CA1998",
             dataType: "jsonp",
             jsonp: "callback",//传递给请求处理程序或页面的，用以获得jsonp回调函数名的参数名(一般默认为:callback)
             jsonpCallback:"flightHandler",//自定义的jsonp回调函数名称，默认为jQuery自动生成的随机函数名，也可以写"?"，jQuery会自动为你处理数据
             success: function(json){
                 alert('您查询到航班信息：票价： ' + json.price + ' 元，余票： ' + json.tickets + ' 张。');
             },
             error: function(){
                 alert('fail');
             }
         });
     });
     </script>
     </head>
  <body>
  </body>
 </html>

是不是有点奇怪？为什么我这次没有写flightHandler这个函数呢？而且竟然也运行成功了！
这就是jQuery的功劳了，jquery在处理jsonp类型的ajax时（但是实际上，jsonp压根不是通过ajax来实现的），自动帮你生成回调函数并把数据取出来供success属性方法来调用。

说道这儿，很多人也许就恍然大悟了。JSONP名字听着高端，其实也不过如此嘛。
